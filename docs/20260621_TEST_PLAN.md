## Overview

| Layer | What it covers | Cost | Speed | When it runs |
|-------|---------------|------|-------|--------------|
| **L1 Unit** | Code in isolation, no I/O | $0 | <30s | Every PR |
| **L2 Integration** | Local mock servers, OAuth round-trips | $0 | <2 min | Every PR |
| **L3 Sandbox** | Real SDKs + schema validators, no network | $0 | <1 min | Every PR |
| **L4 Live API** | Real LLM picks the right tool — no workflow run | ~$0.05/run | ~3 min | Nightly |
| **L5 E2E** | Full surface install + real BioMate workflow run | $0.05–$2/run | 5–20 min | Pre-release + weekly |

## Platform integration patterns — interface-based vs API-based

Every surface falls into one of two integration models. The model determines what needs tuning, what can break silently, and how tests differ.

### Model A — Protocol-based (MCP stdio / SSE)

These surfaces speak MCP JSON-RPC directly to `biomate_mcp_server.py`. The LLM picks tools from the `tools/list` manifest. **No per-surface message formatting exists** — the tool schema IS the interface.

| Surface | Protocol | Config file | What can break |
|---------|----------|-------------|----------------|
| Claude Code | MCP stdio | `~/.claude.json` | Schema change breaks tool selection; config key collision; path regression |
| Claude Desktop | MCP stdio | `~/Library/Application Support/Claude/claude_desktop_config.json` | Same as Claude Code; platform path differs |
| Cursor | MCP stdio | `~/.cursor/mcp.json` | Same as Claude Code; file path differs |
| Codex CLI | MCP stdio | `~/.codex/config.toml` | TOML format (not JSON); `[mcp_servers.biomate]` section |

**Test strategy for Model A:**
- **Schema conformance is the primary risk.** All 17 tools must serialize to both Anthropic and OpenAI SDK formats without `TypeError`. Covered by `test_connector_sandbox.py`.
- **Installer path accuracy.** Each surface writes to a different config file. The installer integration tests (L2 `flow.test.ts`) must assert the exact file path per target.
- **No formatting tuning needed.** The LLM renders tool results as it sees fit. Surface-specific rendering (markdown, inline images) is handled by the host app, not by BioMate.
- **Auth refresh is the only runtime concern.** Silent PKCE refresh must work on all four clients. Covered by scenarios 1.8 / 2 / 3.12 / 4.11 in human test sections above.

**Unique test scenarios per Model A surface:**

| Surface | Extra scenario | Why it matters |
|---------|---------------|----------------|
| Claude Code | `1.3` reinstall idempotency | `mcpServers` JSON merge must not create duplicate keys |
| Claude Desktop | `2.11` inline image render | Desktop renders tool images inline — Code does not |
| Cursor | `3.11` "Add to output" context | Cursor-specific workflow adds tool output to file context |
| Codex | `4.11` long-poll progress | Codex has no streaming UI; `get_run` polling at 10s interval is the only progress signal |

### Model B — Interface-based (REST / webhook adapters)

These surfaces communicate through platform-specific APIs (REST webhooks, HTTP callbacks, platform SDKs). BioMate has a **per-surface adapter** that formats responses into the platform's native message format. **Interface tuning is required** whenever the platform's card schema, message length limit, or button API changes.

| Surface | Integration model | Adapter file | Interface-specific elements |
|---------|------------------|--------------|-----------------------------|
| ChatGPT Custom GPT | OpenAPI 3.1 action spec + OAuth | `connectors/chatgpt/openapi.json` | GPT editor schema import, `operationId` = tool name, `<= 1024 char` descriptions |
| Open Claw / WeChat | HTTP webhook + HMAC | `connectors/wechat/open_claw.py` | WeChat message XML/JSON format, push notification API, in-app browser deep links |
| Slack | Events API + slash commands + Block Kit | `connectors/slack/slack_bot.py` | Block Kit JSON layout, thread message updates (not new messages), interactive components |
| Feishu (Lark) | Event webhook + card API | `connectors/feishu/feishu_bot.py` | Feishu card templates (`card_builder.py`), magic auto-login link, event dedup |
| Telegram | Bot API webhook | `connectors/telegram/telegram_bot.py` | 4096-char message splits, inline keyboard buttons, `send_message` vs `edit_message` |

**Test strategy for Model B:**
- **Message formatting is a first-class test target.** Unlike Model A, a silent formatting bug produces a broken card or truncated message visible to end users but not caught by any L1/L2 test.
- **Platform schema validators.** ChatGPT openapi.json must validate against the GPT editor's schema checker; Slack Block Kit JSON must pass the Block Kit Builder linter. These are L3 sandbox tests.
- **HMAC/signature verification.** WeChat, Slack, and Telegram all sign inbound webhooks. A broken HMAC check lets spoofed events through. Covered by security scenarios S.10 and unit tests.
- **Per-surface rendering tests.** Each adapter must be tested with a real-length BioMate response (long structured text) to verify truncation, card splitting, and fallback behavior.
- **Event dedup.** Feishu and WeChat retry webhooks on timeout. Dedup by `event_id` / `msg_signature` must be tested at L1 (already in `test_feishu_bot.py`).
- **OAuth is per-surface.** ChatGPT uses the shared OAuth server, but the action spec's `securitySchemes` section must reference the correct `authorizationUrl` and `tokenUrl`. Drift between the OpenAPI spec and the live server is a test gap (see L3 gap `test_chatgpt_openapi_matches_manifest`).

**Interface tuning checklist — run when platform SDK/schema changes:**

| Surface | When to re-tune | What to check |
|---------|----------------|---------------|
| ChatGPT | When `openapi.json` spec version bumps or OpenAI changes GPT action schema | Import spec in GPT editor → zero errors; all 17 `operationId`s resolve |
| WeChat / Open Claw | When WeChat platform updates push notification or message API | `/connect` flow, push arrive within 30s, in-app browser deep link |
| Slack | When Block Kit version updates or Slack changes interactive components API | Block Kit JSON validates; buttons render; thread edit replaces (not duplicates) message |
| Feishu | When Feishu card API or event webhook format changes | Cards render with correct field layout; event `schema` version header matches |
| Telegram | When Bot API version updates (especially inline keyboard schema) | Message split at ≤4096 chars; buttons dispatch correct callback_data |

### Platform-scenario cross-reference

Different use cases interact differently with each model:

| Use case | Model A behavior | Model B behavior | Interface tuning needed |
|----------|-----------------|-----------------|------------------------|
| ADMET screening (short SMILES list) | LLM formats as markdown table | Adapter formats as platform card (Block Kit / Feishu card / WeChat text) | Card column widths, mobile truncation |
| RNA-seq (multi-phase, 20 min) | LLM receives `get_run` updates every ~10s and narrates progress | Adapter edits a single thread message in place (Slack) or sends sequential updates (Telegram) | Thread edit vs new message; phase card ordering |
| WGS / CryoEM (large file outputs) | LLM shows S3 URLs; user downloads externally | Adapter cannot attach files >50MB; summarizes with deep link | File size limit + fallback to URL |
| Auto-loop QC remediation | LLM presents `was → now` diff inline | Adapter must render 2-column diff table in platform card format | Card column support; mobile card height |
| GEO accession fetch (30s LLM call) | LLM holds context while `resolve_accession` runs | Adapter must handle platform timeout (Slack: 3s for slash command) by sending immediate ACK then editing | Immediate 200 ACK before tool completes |
| IND §2.6.1 DOCX download | LLM provides S3 signed URL | Adapter attaches file where supported (Slack, ChatGPT), URL elsewhere | File attach API per surface; URL fallback |
| Mandarin-language prompt | LLM answers in Chinese automatically | Adapter sends response as-is (no translation); WeChat/Feishu prefer GB18030-safe content | Encoding check; card character limits |

### When to write interface-based vs API-based tests

Use this decision tree before writing any new test:

```
Is this testing message formatting, card layout, button rendering,
platform-specific character limits, or webhook signature format?
  → Interface-based test (L1 adapter unit or L5 human checklist)
  → Mock the platform webhook; assert the output message structure

Is this testing tool selection, parameter extraction, schema correctness,
OAuth flow, or S3/BioMate API interaction?
  → API-based test (L1 manifest, L2 MCP server, L3 sandbox, L4 live)
  → No platform-specific code involved

Does the test need a real platform account (GPT store, WeChat, Slack workspace)?
  → L5 human checklist scenario (cannot be automated without platform credentials)
  → Document in the human interaction plan above
```
